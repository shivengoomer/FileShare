PROJECT OVERVIEW
================================================================================

PROJECT NAME: FileShare
TAGLINE: QR-based Real-time P2P File Sharing with Temporary Secured Rooms

DESCRIPTION:
FileShare is a full-stack web application that enables secure, temporary file 
sharing through QR codes, passwords, and one-time codes. It combines client-side
P2P file transfers with server-side storage, providing fast peer-to-peer sharing
when peers are online and server fallback for persistence.

TARGET USERS:
- Teams needing quick, secure file transfers
- Remote workers sharing large files
- Temporary project collaboration
- Organizations wanting ephemeral file sharing without permanent storage

DATABASE SCHEMA
================================================================================

TABLE: rooms
  id (UUID, PK)
  created_at (DateTime)
  expires_at (DateTime)
  active_users (Int)
  password_hash (String, nullable) - SHA-256 if password protected
  is_active (Boolean)
  Relationships:
    - files (1:N) - cascade delete
    - otps (1:N) - cascade delete

TABLE: files
  id (UUID, PK)
  room_id (UUID, FK)
  filename (String) - sanitized
  original_filename (String) - user-provided
  size (BigInt)
  content_type (String)
  checksum (String, nullable) - SHA-256 hex
  uploaded_at (DateTime)

TABLE: room_otps
  id (UUID, PK)
  room_id (UUID, FK)
  otp_code (String) - 6 digits
  created_at (DateTime)
  expires_at (DateTime)
  consumed (Boolean) - single-use

Indexes:
  - room_id on files (for lookups)
  - expires_at on rooms (for cleanup)
  - otp_code on room_otps (for validation)


API ENDPOINTS
================================================================================

ROOM MANAGEMENT:

POST /rooms/create
  Body: { expiry_minutes: int, password?: string }
  Response: { id, created_at, expires_at, active_users, has_password, file_count }
  Purpose: Create a new temporary sharing room

GET /rooms/{room_id}
  Response: Room metadata
  Purpose: Fetch room details without joining

POST /rooms/{room_id}/join
  Body: { password?: string }
  Response: Room metadata
  Purpose: Join a password-protected room

POST /rooms/{room_id}/join/otp
  Body: { otp: string }
  Response: Room metadata
  Purpose: Join room using OTP (requires room_id)

POST /rooms/join/otp
  Body: { otp: string }
  Response: Room metadata including id
  Purpose: Join ANY room with only OTP (no room_id needed)

POST /rooms/{room_id}/otp/generate
  Response: { otp_code, expires_in_seconds, expires_at }
  Purpose: Generate single-use OTP (only room owner)

GET /rooms/{room_id}/qr
  Response: PNG image
  Purpose: Get QR code for room join link

DELETE /rooms/{room_id}
  Purpose: Manually end room, delete all files


FILE OPERATIONS:

POST /rooms/{room_id}/upload
  Body: multipart/form-data (files)
  Response: [{ id, filename, size, checksum }]
  Purpose: Simple single-request file upload

POST /rooms/{room_id}/files/chunked/init
  Body: { filename, size, content_type }
  Response: { upload_id, chunk_size }
  Purpose: Initialize chunked upload

PUT /rooms/{room_id}/files/chunked/{upload_id}/{chunk_index}
  Body: binary chunk data
  Response: { chunk_index, crc32 }
  Purpose: Upload single chunk with CRC verification

POST /rooms/{room_id}/files/chunked/{upload_id}/complete
  Body: { sha256 }
  Response: { id, filename, size, checksum }
  Purpose: Finalize and verify full file

GET /rooms/{room_id}/files
  Response: [{ id, filename, size, content_type, checksum, uploaded_at }]
  Purpose: List all files in room

GET /rooms/{room_id}/files/{file_id}
  Response: File content (streaming)
  Headers: content-disposition, x-file-checksum
  Purpose: Download single file

GET /rooms/{room_id}/files/{file_id}/zip
  Response: ZIP archive
  Purpose: Download all files as ZIP

GET /rooms/{room_id}/files/{file_id}/verify
  Response: { verified: bool, checksum }
  Purpose: Download with server-side verification


WEBSOCKET ENDPOINT:

WS /ws/{room_id}
  Purpose: Real-time room events and P2P signaling

Messages sent FROM server TO client:
  {
    type: "hello",
    peer_id: "uuid" - your unique ID in this room
  }
  {
    type: "user_count",
    count: int - current active users
  }
  {
    type: "user_joined"
    - triggered when new peer joins (other peers get this)
  }
  {
    type: "file_available",
    file: { id, original_filename, size, content_type, peer_id }
  }
  {
    type: "file_uploaded",
    file: { id, original_filename, size, content_type }
  }
  {
    type: "user_left",
    peer_id: "uuid" - disconnected peer
  }
  {
    type: "room_expired"
    - room no longer active
  }
  {
    type: "file_request",
    from: "peer_id",
    file_id: "uuid",
    conn_id: "uuid"
  }
  {
    type: "webrtc_offer",
    from: "peer_id",
    conn_id: "uuid",
    sdp: string
  }
  {
    type: "webrtc_answer",
    conn_id: "uuid",
    sdp: string
  }
  {
    type: "webrtc_ice",
    conn_id: "uuid",
    candidate: string
  }

DATA SECURITY
================================================================================

ENCRYPTION & HASHING:
  ✓ Passwords: SHA-256 (one-way hash)
  ✓ Files: SHA-256 checksum verification
  ✓ Transport: HTTPS/WSS recommended in production
  ✓ P2P: Direct peer-to-peer (encrypted by browser's WebRTC)

PRIVACY MEASURES:
  ✓ No permanent user accounts
  ✓ Files auto-deleted on room expiry
  ✓ No audit logs of file contents
  ✓ No metadata beyond room lifetime
  ✓ OTP single-use (consumed immediately)
  ✓ No file versioning or history

ACCESS CONTROL:
  ✓ Room ID required (UUID - not guessable)
  ✓ Password hash checked (if enabled)
  ✓ OTP validated & consumed (single-use)
  ✓ WebSocket per-peer ID (not replayable)
  ✓ File extensions blocked (exe, bat, sh, ps1, etc.)

NETWORK SECURITY:
  ✓ CORS protection (configurable origins)
  ✓ File size limits (configurable)
  ✓ Upload rate limiting (can be added)
  ✓ Input sanitization (filenames)
  ✓ WebSocket origin validation

DEPLOYMENT GUIDE
================================================================================

LOCAL DEVELOPMENT:
  1. Clone repository
  2. Backend setup:
     cd backend
     python -m venv .venv
     source .venv/bin/activate
     pip install -r requirements.txt
     # Create .env file
     python -m backend.main
  3. Frontend setup:
     cd frontend
     npm install
     npm run dev
  4. Open http://localhost:5173

DOCKER DEPLOYMENT:
  # Build both images
  docker build -t fileshare-backend ./backend
  docker build -t fileshare-frontend ./frontend
  
  # Run with docker-compose
  docker-compose up

AZURE APP SERVICE:
  1. Create PostgreSQL database
  2. Deploy backend via:
     az webapp up --name fileshare-backend --resource-group mygroup
  3. Deploy frontend via:
     az webapp up --name fileshare-frontend --resource-group mygroup
  4. Run startup.sh for initialization

CONFIGURATION (via .env):
  DATABASE_URL=postgresql+asyncpg://user:pass@host:5432/dbname
  SECRET_KEY=your-secret-key
  ROOM_EXPIRY_MINUTES=30
  MAX_FILE_SIZE_MB=100
  UPLOAD_DIR=./uploads
  CORS_ORIGINS=["http://localhost:5173"]

USE CASES & BENEFITS
================================================================================

USE CASE 1: Design Team Collaboration
  - Designer uploads high-res files to secure room
  - Team joins via QR code scan
  - Discuss in real time
  - Download files automatically deleted after 1 hour
  BENEFITS: No permanent storage, instant sharing, real-time sync

USE CASE 2: Contractor File Transfer
  - Client creates room with OTP
  - Shares OTP to contractor (via email/call)
  - Contractor joins, uploads completed work
  - Client downloads files
  - Room expires automatically (30 min)
  BENEFITS: Secure guest access, no account needed, auto-cleanup

USE CASE 3: Multi-Office File Exchange
  - Office A uploads files to P2P room
  - Office B peers join same room
  - Direct P2P transfer (no server bandwidth used)
  - If Office B is offline, files persist on server until expiry
  BENEFITS: Bandwidth efficient, P2P when available, server fallback

USE CASE 4: Confidential Document Sharing
  - Legal team creates password-protected room
  - Generates OTP for client access
  - Client joins, downloads docs
  - Room deleted manually when done
  - Files never stored after deletion
  BENEFITS: Maximum control, audit trail possible, no data trail


ADVANTAGES OVER ALTERNATIVES
================================================================================

vs. Email:
  ✓ No file size limits (email: 25MB typical)
  ✓ Real-time collaboration
  ✓ No permanent inbox storage
  ✓ Faster for large files
  ✗ Requires both parties online for P2P

vs. Cloud Storage (Google Drive, Dropbox):
  ✓ No account signup required
  ✓ Automatic deletion (privacy)
  ✓ Cheaper (ephemeral vs. persistent storage)
  ✓ Lower latency (P2P direct)
  ✗ No version history
  ✗ No UI collaboration tools

vs. File Transfer Services (WeTransfer, Tresorit):
  ✓ Self-hosted option
  ✓ Longer expiry options (up to 60 minutes)
  ✓ P2P capability (no server relay needed)
  ✓ Open source (transparency)
  ✗ Smaller feature set
  ✗ No advanced sharing controls

vs. Messaging Apps (Slack, Teams):
  ✓ Larger file support
  ✓ Temporary by design
  ✓ Dedicated file sharing
  ✗ Requires messaging platform account

TROUBLESHOOTING GUIDE
================================================================================

SYMPTOM: "Room not found"
  CAUSE: Room ID incorrect or room expired
  FIX: Create new room or get correct ID from QR

SYMPTOM: "Password incorrect"
  CAUSE: Wrong password entered
  FIX: Ask room creator for correct password

SYMPTOM: "OTP invalid or expired"
  CAUSE: OTP wrong or exceeded 5-min window
  FIX: Room owner generates new OTP

SYMPTOM: "Upload failed"
  CAUSE: File too large, room expired, or network error
  FIX: Check file size, refresh room, retry

SYMPTOM: "P2P transfer failed"
  CAUSE: Peer offline, firewall blocking WebRTC
  FIX: Server fallback should kick in automatically

SYMPTOM: "WebSocket connection lost"
  CAUSE: Network interruption
  FIX: Page will auto-reconnect; refresh if needed

SYMPTOM: "Room expired banner appears"
  CAUSE: Room's expiry time reached
  FIX: Create new room; old files auto-deleted
